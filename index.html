Saer field guide

Every capability an agent gains at the edge is also a governance surface.

When an agent reaches a phone, it can see, hear, speak, locate, and render. The interesting question is not what it can do — it is what it is allowed to do, where, and under whose permission. This is the full agent node command surface, and exactly how each command is gated.

Built on OpenClaw, the open-source agent gateway. Every command, platform, and gate below is verified against its node command policy and the iOS and Android node clients. Full reference: docs.openclaw.ai.

How access is decided

Two gates stand between an agent and a device.

A node advertises a command surface; the gateway decides what an agent may actually invoke. A command runs only when both gates pass — which means capability and permission are separate facts, by design.

Gate 01

Declaration

The device lists the command during its handshake — and only when the matching capability is enabled and the OS permission is granted. The phone decides what it is willing to offer.

Gate 02

Gateway policy

The platform allowlist must permit the declared command. allowCommands adds capability; denyCommands removes it and always wins. The operator decides what is permitted.

Gating classes

Not all capability is equal.

D Default

Allowed out of the box for the platform. The device still needs the capability enabled and the OS permission granted.

O Opt-in · high risk

Blocked until explicitly added to allowCommands: camera capture, screen recording, contact and calendar writes, SMS.

— Not exposed

The platform does not offer the command at all. Absence is the strongest gate.

Capability gated

Advertised only when the user enables the feature, or the hardware supports it.

Permission gated

Returns an error until the matching OS permission is granted on the device.

Trusted talk

Push-to-talk is allowed by default only for nodes that advertise the talk capability.

The matrix · desktop and headless

The same protocol, a different body.

A node is a companion device, not a special mobile layer. The macOS app in node mode carries the richest surface; a headless host carries almost none.

macOS node mode

The menubar app acting as a node exposes the broadest surface:

  • canvas.*, camera.list (default) with camera.snap / camera.clip (opt-in)
  • location.get, a real screen.snapshot of the display, plus screen.record (opt-in)
  • system.run / system.which, exec approvals, and browser.proxy

Exec, browser-proxy, and screen snapshot are desktop-host commands: held back until the node is connected and declared, and system.run is further gated by exec approvals (ask / allowlist / full).

Headless node host

openclaw node run on Linux, Windows, or macOS exposes only:

  • system.run and system.which
  • system.execApprovals.get / system.execApprovals.set
  • plus any plugin-provided node-host commands

No canvas, no camera, no sensors. When a body has no senses, the gate is the absence of the command itself.

Foreground and background

A phone is not an always-on server.

The platform itself is a gate. The OS, not the operator, decides when an agent may reach through.

iOS

  • Requires the app foregrounded for canvas.*, camera.*, screen.*, and talk.*. Background invokes return NODE_BACKGROUND_UNAVAILABLE.
  • location.get works in the background only when the location permission is set to Always; otherwise LOCATION_BACKGROUND_UNAVAILABLE.
  • Voice wake is a separate capability and is best effort while backgrounded.

Android

  • Requires the foreground for canvas.* (including A2UI) and camera.*.
  • Other commands, including talk.ptt.*, run in the background — a deliberate difference from iOS.
  • Personal-data commands stay subject to OS permission and the platform wake budget.
When a gate refuses

The error codes are the governance, made legible.

command not allowlisted / not declared by nodeOne of the two gates failed. Check the platform allowlist and the node command list.
NODE_BACKGROUND_UNAVAILABLEBring the node app to the foreground.
LOCATION_BACKGROUND_UNAVAILABLESet the location permission to Always for background reads.
LOCATION_DISABLED / LOCATION_PERMISSION_REQUIREDEnable Location in Settings and grant the OS permission.
*_PERMISSION_REQUIREDGrant the matching OS permission (camera, SMS, and so on) on the device.
SYSTEM_RUN_DENIEDThe exec approval policy rejected the run.
Get started

Connect your iOS node in five steps.

A phone is a node, not a gateway: it connects to a gateway you run, and pairing is approved by you, never bypassed. That approval is Gate 02 in practice.

You will need: a machine running the OpenClaw gateway (macOS, Linux, or Windows via WSL2) reachable from the phone over the same LAN or a tailnet, plus the OpenClaw iOS app.
1

Run a gateway

On your Mac, Linux box, or WSL2. The phone connects to this.

openclaw gateway --port 18789
2

Install the app

Via TestFlight or an Apple build when a release enables it, or build from source from apps/ios in Xcode.

3

Discover and connect

In the app Settings, pick a gateway found on your LAN (Bonjour) or tailnet, or enable Manual Host and enter host:port (default 18789).

4

Approve the pairing

On the gateway host, approve the device. Nothing connects until you do.

openclaw devices list openclaw devices approve <requestId>
5

Verify

Confirm the node is paired and online. The matrix commands are now invokable, subject to the gating above.

openclaw nodes status

Full install, discovery, and troubleshooting: the OpenClaw iOS app guide.

The Saer lens

This is one device. Now multiply it by every agent, tool, and node you run.

The matrix above governs a single phone, through one gateway, with a handful of config keys. That model holds for one device. It does not hold for a company where dozens of agents touch tools, data, memory, and decisions — each with its own allowlist, its own approvals, its own logs. Saer is building the layer where this kind of governance is native, not re-invented per agent: permission, evidence, review, and consequence as one operating surface.

Two gates → permissionCapability and authority kept as separate facts, everywhere — not just on one device.
Opt-in & deny → reviewHigh-stakes actions become proposed state until the right authority accepts them.
Error codes → consequenceA refusal carries a reason. Every action carries intent, evidence, and outcome.
